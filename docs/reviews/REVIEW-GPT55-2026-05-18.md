## Findings

### Blockers

- **B-01 — Plugin auto-starts arbitrary MCP servers discovered from other editors** (`src/main.ts:160`) — `McpDiscovery.discoverAllConfigs()` imports MCP configs from VS Code, Cursor, Claude Desktop, Windsurf, JetBrains, etc., and `this.mcpManager.start(this.toMcpServerConfigs(discovered))` immediately starts every discovered server with `autoStart !== false`. This means enabling the Obsidian plugin can execute arbitrary stdio commands configured for another app without explicit Obsidian consent. Fix: do not auto-start discovered MCP servers; import them disabled, show them in settings, and require an explicit per-server enable/start decision before spawning stdio or connecting HTTP/SSE transports.

- **B-02 — Runtime tool calls lose annotations before permission evaluation** (`src/main.ts:79`) — `DeferredProvider.toToolCall()` creates `ToolCall` objects with only `id`, `name`, `serverName`, `arguments`, and `status`, so `PermissionGate.evaluate()` receives `toolCall.annotations === undefined`. Because `PermissionGate` defaults absent annotations to `destructiveHint: true` and `openWorldHint: true`, real read-only tools are misclassified and may be denied or shown as destructive/network tools. Fix: preserve annotations by resolving streamed tool-call names against the native/MCP tool registry before permission evaluation, or include a registry lookup inside the permission adapter.

- **B-03 — Streaming tool-call deltas are treated as complete calls instead of being accumulated** (`src/providers/openaiAdapter.ts:157`) — OpenAI-compatible streaming sends tool call IDs, names, and JSON arguments across multiple deltas, but `mapChunk()` emits each delta independently and `ChatViewModel` simply appends `chunk.toolCalls` into `toolCalls`. Partial JSON like `{"q"` is parsed as `{ value: rawArgs }`, causing malformed or duplicate tool calls. Fix: accumulate deltas by `index`/`id` until `finish_reason === "tool_calls"`, concatenate `function.arguments`, preserve the final name/id, then emit complete tool calls only once.

### High

- **H-01 — "Always allow this tool" is silently downgraded to conversation-only and never persisted** (`src/chat/ChatView.ts:123`) — `toChatConsentDecision()` maps `allow-forever` to `"allow-session"`, and there is no code that writes persistent `toolPolicies`/`nativeToolPolicies`. Users who select "Always allow this tool" are misled, and permission behavior resets unexpectedly. Fix: either remove the "Always allow" UI or persist the decision into settings keyed by qualified tool name and save settings immediately.

- **H-02 — "Allow for this conversation" is not remembered for later calls** (`src/chat/ChatViewModel.ts:166`) — `ConsentDecision` includes `allow-session`, and `PermissionGate.evaluate()` has session-allow support in the pure API, but `ChatViewModel` never records allowed tool names after consent. Every future call asks again, and the permission state machine does not match the consent UI. Fix: maintain a per-conversation/session allowed set and add the qualified tool name when consent is `allow-session`.

- **H-03 — User-configured Azure endpoints are accepted without HTTPS validation** (`src/providers/AzureFoundryProvider.ts:36`) — Azure Foundry and Azure OpenAI Classic pass user-provided endpoints through `normalizeEndpoint()` without checking scheme, so `http://` or attacker-controlled non-Azure endpoints can receive prompts, vault context, and API keys. Fix: validate provider endpoints before construction: require `https:`, reject credentials in URLs, and warn/require explicit opt-in for nonstandard hosts.

- **H-04 — MCP HTTP/SSE URLs are accepted without HTTPS validation** (`src/mcp/McpDiscovery.ts:211`) — discovered and configured MCP URLs are converted with `new URL(transport.url)` later, but no policy blocks `http://` or other unsafe schemes. This can send MCP headers/tokens and vault-derived tool data over plaintext or to an unexpected endpoint. Fix: reject non-HTTPS remote MCP URLs by default; allow `http://localhost` only with explicit local-development consent.

- **H-05 — Audit settings are exposed but ignored by the audit writer** (`src/security/AuditLogger.ts:5`) — `AuditLogger` hard-codes `.obsidian/plugins/github-copilot-agent/audit.jsonl` and `10MB`, while settings define `auditLogEnabled`, `auditLogPath`, and `auditLogMaxSizeMb`. Disabling audit logging or changing its path/size in settings has no effect, which breaks user expectations and compliance controls. Fix: pass audit settings into `AuditLogger`, honor disabled state, validate vault-relative paths, and use configured rotation size.

### Medium

- **M-01 — Native vault tools rely on Obsidian path lookup without explicit traversal/canonicalization checks** (`src/chat/nativeTools.ts:76`) — `read_vault_file`, `create_note`, `append_note`, and `delete_note` accept model-supplied paths directly. Even if Obsidian usually scopes vault APIs, the plugin should explicitly reject absolute paths, drive letters, UNC paths, `~`, and `..` before read/write/delete. Fix: add a shared `validateVaultRelativePath()` used by all native tools and attached-file reads.

- **M-02 — Trusted content defaults include `Templates`, treating common attacker-editable notes as trusted instructions** (`src/settings/settings.ts:80`) — `trustedFolders` defaults to `["Trusted", "Templates"]`, and `wrapForLlm()` omits untrusted wrappers for trusted paths. Templates are commonly copied/imported and may contain arbitrary markdown; trusting them by default weakens prompt-injection isolation. Fix: default `trustedFolders` to `[]` or a plugin-created folder with clear onboarding, and require explicit user opt-in for `Templates`.

- **M-03 — Audit redaction misses common secret value patterns when keys are innocuous** (`src/security/AuditLogger.ts:8`) — redaction only triggers on key names matching `key|token|secret|password|api[-_]?key|authorization|credential`. Values like `Bearer ...`, `github_pat_...`, `gho_...`, `ghu_...`, `ghs_...`, or `tid=...` under keys such as `value`, `content`, or `args` can persist to audit logs. Fix: redact both by sensitive key names and by sensitive value patterns/high-entropy token prefixes recursively.

- **M-04 — Tool results are fed back to the model without untrusted-content wrapping** (`src/chat/ChatViewModel.ts:180`) — native and MCP tool results are inserted as plain tool messages. The system prompt says tool results are untrusted, but the actual result text is not wrapped or delimited like note content. Malicious MCP output can look like direct instructions. Fix: wrap tool result text in an explicit untrusted/tool-result envelope before adding it to provider messages, preserving raw text only for UI display.

### Low

- **L-01 — `classicApiVersion` setting is ignored** (`src/providers/AzureOpenAIClassicProvider.ts:39`) — settings expose `classicApiVersion`, but the provider hard-codes `"2024-10-21"`. Users cannot change API version even though the UI implies they can. Fix: add `apiVersion` to `AzureOpenAIClassicProviderConfig` and pass `settings.classicApiVersion` through the factory.

- **L-02 — `_pending-imports.d.ts` was not present** (`src/`) — verified no `_pending-imports.d.ts` match exists. No fix required; keep it from reappearing in CI with a simple grep check if desired.

- Device-flow and Copilot token exchange endpoints are hard-coded to HTTPS GitHub endpoints.
